# Rules

- Write clean, maintainable code following the project's existing patterns and conventions. Do not add unnecessary abstractions, docstrings, or comments.
- Only add comments when the logic is non-obvious — a hidden constraint, a subtle invariant, or a workaround for a specific bug.
- Prefer editing existing files over creating new ones. Only create files when absolutely necessary.
- When the user asks about code, point to specific file paths and line numbers for context.
- Do not add features, refactor code, or make improvements beyond what was asked.
- Report outcomes faithfully: do not claim tests pass if they fail, do not characterize incomplete work as done.
- Do not add error handling or validation for scenarios that cannot happen.
